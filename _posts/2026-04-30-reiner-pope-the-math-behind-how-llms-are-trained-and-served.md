## 一、这期核心在讲什么（先给结论）
- 问题：为什么“更贵的 Fast 模式”能换来更低延迟？  
  观点：关键杠杆是 batch size（批处理大小），其次才是 speculative decoding / 多 token 预测。  
  结论：速度和价格不是线性互换，主要受“是否把固定成本摊薄”控制。
- 问题：能不能无限加钱换无限低延迟？  
  观点：给定硬件配置存在“延迟下限”，受权重读取带宽限制。  
  结论：再加价也会撞到物理上限，不可能无限快。
- 问题：慢一点能不能便宜很多（Slow Mode）？  
  观点：当成本已接近计算下限后，再慢也很难继续降太多。  
  结论：慢模式的降价空间有限，尤其在已充分 batch 的区间。
- 问题：长上下文为什么贵、且可能拖垮效率？  
  观点：KV cache 访问随上下文增长（在稠密注意力下近似线性）并可能把系统推向 memory-bound。  
  结论：长上下文不仅增成本，还可能显著降低硬件利用率（MFU）。
- 问题：能否从公开信息反推实验室工程策略？  
  观点：用“公式 + API 价格 +硬件常识”可做逆向估计。  
  结论：即便不开源内部细节，外部也能推断架构与服务策略的大方向。

## 三、关键概念白话版
1. Batch Size  
是什么：一次并行服务多少请求/序列。  
为什么重要：它决定固定内存读取成本能否被摊薄。  
常见误区：以为 batch 只影响吞吐，不影响单 token 成本。

2. Active Parameters vs Total Parameters  
是什么：MoE 等模型每 token 只激活部分参数（active），但权重系统仍有总量（total）。  
为什么重要：计算压力更看 active，内存搬运常受 total 影响。  
常见误区：把“总参数大”直接等同于“每 token 计算一定更贵”。

3. KV Cache  
是什么：生成下一个 token 时，历史 token 的键值状态缓存。  
为什么重要：长上下文时 KV 读取会迅速放大内存带宽压力。  
常见误区：以为上下文成本只体现在“多算几步注意力”，忽略了“搬数据”更贵。

4. Roofline（屋顶线）分析  
是什么：用硬件算力上限与带宽上限判断系统瓶颈。  
为什么重要：可快速判断某优化是“算力有效”还是“带宽有效”。  
常见误区：只看 FLOPS，不看内存带宽与数据移动路径。

5. Memory-bound / Compute-bound  
是什么：分别由内存访问或计算单元成为主要瓶颈。  
为什么重要：两种状态的最优工程策略完全不同。  
常见误区：认为买更强 GPU 就能同时解决两类瓶颈。

6. Cost per Token（近似 `T/B`）  
是什么：每 token 的资源成本近似由单次耗时除以并行批数得到。  
为什么重要：直接对应 API 定价下限与毛利空间。  
常见误区：把延迟下降简单等同于成本上升或下降，忽略区间差异。

## 四、我们怎么看
- 判断 1：如果模型服务已运行在“计算托底区”，那么再推慢速档，价格下降不会显著。  
依据：cost 曲线在大 batch 区间趋于平台。  
影响对象：面向高并发的 API 厂商、企业级包量客户。  
演化：短期会看到“档位更多”，中期会回归“差异化能力收费”而非单纯慢换便宜。

- 判断 2：如果上下文继续拉长而注意力机制不变，内存带宽会成为更硬的商业约束。  
依据：KV fetch 随 context 上升，易把系统从 compute-bound 推到 memory-bound。  
影响对象：主打超长上下文的产品线、Agent 场景。  
演化：中期会加速稀疏注意力、KV 压缩、分层记忆等方案落地。

- 判断 3：如果外部能持续拿到 API 价格、延迟与硬件代际信息，就能反推出厂商架构策略的大方向。  
依据：本期展示了“少量方程 + 公共价格”即可做工程逆向。  
影响对象：投资、采购、竞品分析团队。  
演化：短期是粗粒度推断，随着公开 benchmark 与价格细分增多，推断会更精确。
