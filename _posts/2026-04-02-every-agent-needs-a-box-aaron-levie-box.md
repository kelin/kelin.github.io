这期对谈发生在一个很关键的时间点：一边是“AI 会不会杀死 SaaS”的舆论拉扯，另一边是 Box 这类老牌 SaaS 仍在增长并把重心压到 Agent 基础设施。节目链接：https://www.latent.space/p/box 。

### 嘉宾速览（如有）
- Aaron Levie：Box 联合创始人/CEO，长期做企业内容管理、权限与协作平台。
- 代表背景：其业务覆盖约 67%（节目中口径）Fortune 500 客户，兼具“上市公司运营”与“前沿 AI 产品讨论”视角。
- 与本期相关性：他讨论的不是“Agent demo”，而是 Agent 在企业真实环境落地时的权限、治理、评测与组织改造问题。
- 联合主持 Jeff Huber：对“context engineering / agentic search”有持续研究，在节目中多次补充技术视角。

## 一、这期核心在讲什么（先给结论）
- 结论 1：**问题不是“Agent 会不会来”，而是“企业是否愿意重写工作流”**。观点是企业要适配 Agent，而非 Agent 先适配旧流程；结论是早改造团队会拿到复利优势。
- 结论 2：**“每个 Agent 都需要一个 Box”本质是“每个 Agent 都需要受控工作空间”**。观点是 Agent 需要文件、权限、身份和审计边界；结论是文件系统+沙箱将成为企业 Agent 基础设施层。
- 结论 3：**Coding Agent 先爆发不是偶然，而是条件最优**。观点是代码场景文本化程度高、反馈快、权限相对集中；结论是通用知识工作 Agent 落地会更慢、更重、更依赖组织治理。
- 结论 4：**上下文工程是当前瓶颈，不是上下文窗口大小**。观点是“从海量文档中选对少量上下文”远比“把所有东西都塞进去”现实；结论是搜索、排序、回溯、放弃策略决定 Agent 可用性。
- 结论 5：**2026 年的主战场会从“写代码”转向“知识工作 Agent”**。观点是法务、金融、公共部门等会进入 Agent 化；结论是评测体系、DevRel 和实施能力会成为新竞争壁垒。

## 二、关键内容拆解（按节目脉络）
- （约 00:00）**工作流重构先于自动化神话**  
  观点：今天的核心变化是人去适配 Agent。  
  证据：Levie 直言“你不再直接写代码，而是让 Agent 去做，你更多在审阅甚至更上层任务”。  
  结论：企业导入 Agent 不是买个工具，而是组织流程变更工程。

- （约 01:29）**“Every Agent Needs a Box”是基础设施命题**  
  观点：企业历史文件在 Agent 时代从“被动存档”变“主动生产资料”。  
  证据：合同、研究、销售、路线图等过去散落文件，开始被 Agent 反复检索和重组。  
  结论：文件系统不再只是存储层，而是 Agent 价值产出的上下文底座。

- （约 04:38）**Agent 身份治理比“给个账号”复杂得多**  
  观点：Agent 不能直接套用人类账号模型。  
  证据：人类员工有隐私边界与个人责任，Agent 当前缺乏等价法律主体，创建者往往要承担责任。  
  结论：企业需要“Agent 身份 + 责任归属 + 监督机制”的新范式。

- （约 04:38-11:28）**未来 Agent 数量会远多于人，安全事故会常态化**  
  观点：Prompt injection、越权读取会成为高频风险。  
  证据：节目明确预判“会出现非常夸张的安全事件”。  
  结论：权限、审计、数据治理不是合规附属，而是 Agent 规模化前提。

- （约 11:28）**为什么 Coding Agent 先起飞**  
  观点：代码工作天然具备高可机器化条件。  
  证据：文本进文本出、数据格式统一、开发者高密度使用带来快速反馈闭环。  
  结论：别把“程序员场景的速度”误判为“全行业同速扩散”。

- （约 11:28-21:42）**企业知识工作落地难在“多重逆风”**  
  观点：企业真实场景有权限割裂、非结构化沟通、跨系统数据碎片。  
  证据：金融/销售流程涉及会话、会议、线下交流，不是纯文本流水线。  
  结论：知识工作 Agent 是多年期工程，不是季度级部署。

- （约 21:42）**上下文工程的数学矛盾**  
  观点：不是“模型不聪明”，而是“信息空间太大”。  
  证据：Levie 用“数千万页信息 vs 有限 token”描述检索难题，并强调“何时停止搜索”是新能力。  
  结论：高质量 Agent 需要“检索-验证-重试-放弃”策略，而非一次性问答。

- （约 31:29）**评测从“测模型”转向“测 Agent 系统”**  
  观点：企业需要同时评估模型能力与执行框架。  
  证据：他们从 one-shot 模型测试，转向更 agentic 的端到端评测，并跟踪回归。  
  结论：没有持续 eval，企业 Agent 质量不可控。

- （约 33:23）**行业数据集差异决定能力上限**  
  观点：法务、医疗、公共部门、金融各自语料和合规压力不同。  
  证据：节目提到按行业设计测试集，比较模型在真实任务的提升。  
  结论：通用 benchmark 只能参考，真正上线靠垂直场景评测。

- （约 35:22）**组织打法：不是“创新中心”，是“生死核心队”**  
  观点：Agent 转型要有跨层协同，但关键决策需集中推进。  
  证据：Levie 描述核心 AI/Agent 团队与搜索、基础设施等同心圆支持结构。  
  结论：大公司做 Agent 需要“核心突破 + 全栈配套”双轨机制。

- （约 38:50）**读写一体化：读难于写，但写的质量感知更苛刻**  
  观点：读取企业正确上下文比生成内容更难。  
  证据：检索错文档会直接失信；而生成 PPT/文档又常出现格式细节错误。  
  结论：企业级 Agent 产品要同时解决“检索正确性”和“交付可用性”。

- （约 01:06:47）**DevRel 和媒体化能力会成为 AI 时代分发引擎**  
  观点：当产品/API 都要被 Agent 选择，内容与开发者沟通能力成为核心。  
  证据：节目多次强调 DevRel 需求上升、公司需要“像媒体一样持续输出”。  
  结论：AI 竞争不仅是模型与产品，也是谁能更快占领开发者与 Agent 注意力。

## 三、关键概念白话版
- **Agent Box（Agent 工作箱）**  
  是什么：给 Agent 的受控文件系统/沙箱与权限边界。  
  为什么重要：没有边界，Agent 只会变成企业安全事故放大器。  
  常见误区：把它理解成“多开几个账号”。

- **Agent Identity（Agent 身份）**  
  是什么：Agent 在企业系统中的身份、权限、责任映射。  
  为什么重要：决定谁能看什么、谁对行为负责、如何审计。  
  常见误区：默认“Agent = 用户本人”。

- **Context Engineering（上下文工程）**  
  是什么：从海量信息中挑选“最该给模型看的那部分”。  
  为什么重要：上下文选错比模型弱一点更致命。  
  常见误区：只盯上下文窗口长度，不管检索与排序质量。

- **Agentic Search（代理式搜索）**  
  是什么：Agent 多轮检索、验证、再检索的动态过程。  
  为什么重要：真实企业知识分散且噪声高，一次检索命中率不够。  
  常见误区：把搜索当静态关键词匹配。

- **Evals（评测体系）**  
  是什么：持续验证 Agent 在真实任务中的准确率、稳定性、回归情况。  
  为什么重要：没有 eval，模型更新和提示词改动会悄悄把系统带偏。  
  常见误区：用一次 demo 成功替代长期质量管理。

- **Read/Write Workflow（读写工作流）**  
  是什么：Agent 既要读企业知识，也要写回产物与中间状态。  
  为什么重要：只读不写难形成闭环，只有写不读又容易“无根生成”。  
  常见误区：把“生成内容”当全部价值，忽视来源可靠性。

- **Token FOMO**  
  是什么：团队担心“算力没跑满就落后”的焦虑文化。  
  为什么重要：会影响组织节奏、成本决策和实验优先级。  
  常见误区：把“跑得多”当“跑得对”。

## 四、我们怎么看
- 我们同意这期最有价值的判断：**企业 Agent 化是“治理工程”，不只是“模型工程”**。依据是节目反复落在身份、权限、责任、审计和跨部门流程，而不是单点模型参数。受影响最大的是中大型企业 IT、法务、安全和业务运营团队。短期（6-12 个月）会先做半自动流程，中期（1-3 年）才可能进入高自治代理。
- 我们认为“Coding 先行、知识工作滞后”会继续成立。依据是编码场景的数据可得性、反馈密度、工具生态都明显更优。直接影响是：非研发部门的 AI 预算会先买“可控提升”，而不是“一步到位替代人”。  
- 我们也看到一个现实张力：Levie 强调文档化与结构化是前提，但企业现实是信息长期脏乱。  
  - 如果企业愿意先做“文档治理 + 权限治理”，那么 Agent 会带来复利。  
  - 如果企业只上前端聊天入口，不清数据和权限，短期会出现“看起来聪明、实际不可信”的反噬。  
  - 如果评测体系能覆盖真实业务任务，模型迭代会是增益；否则会反复回归、用户信任下降。

## 五、一句可直接借鉴
先别急着“全员上 Agent”，先做一件事：把你团队未来 90 天最关键的 3 条工作流，按“可访问数据、权限边界、验收指标”各补齐一版，再让 Agent 接管其中可评测的子任务。
