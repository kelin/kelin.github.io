## 一、这期核心在讲什么（先给结论）
- 长时任务失败的根因不是“模型不够聪明”，而是两类结构性问题：上下文衰减与自评偏差；结论是要改架构，而不只是改提示词。  
- “压缩上下文”只能延寿，不能治本；真正有效的是 `context reset + 结构化工件交接`，让新 Agent 在干净上下文中接力。  
- 主观任务（尤其前端设计）里，生成器给自己打分会偏乐观；必须引入独立评估器，把“审美”转成可评分指标并持续迭代。  
- V1 的 Planner/Generator/Evaluator 三段式把“需求理解-实现-验收”拆开，显著提高可玩性与可修复性，但成本高。  
- V2 随模型能力提升而“去组件化”（去 Sprint、后置评估），证明 Harness 的核心不是固定流程，而是可随模型代际演进的动态架构。  

## 二、关键内容拆解（按节目脉络）
- 问题起点：长任务会“提前收尾”。  
  证据是上下文窗口逐步填满后，模型更倾向保守结束而非继续深挖。  
  结论是长程开发不能只靠单线程对话。

- “上下文焦虑”不是 token 数字问题，而是状态管理问题。  
  即便压缩历史，也只是推迟失效时间。  
  所以状态要外置为结构化产物，而不是埋在聊天记录里。

- `context reset` 的本质是“有损重启 + 无损交接”。  
  清空窗口后，通过规格、任务单、验收标准等工件传递关键状态。  
  结论：这更像软件工程里的交接协议，而非聊天连续性。

- 第二个根因是“自我评估偏差”。  
  在审美/体验类任务上，模型对自己产出常过度乐观，且与人评有系统偏差。  
  结论：评估职责必须与生成职责分离。

- 前端设计实验把主观问题做了工程化。  
  评分维度被拆成设计质量、原创性、工艺、功能性。  
  结论：只有可评分，才有可迭代。

- 评估器不是“看图说话”，而是可操作页面。  
  通过 Playwright MCP 与页面实时交互，不只看静态截图。  
  结论：动态行为验证比静态视觉判断更接近真实用户体验。

- 迭代策略是 5-15 轮“评分驱动”的探索。  
  每轮按分数选择“深化当前方向”或“转向新方案”。  
  结论：这是把创意搜索过程显式化，而非一次性生成。

- 典型跃迁案例出现在约第十轮。  
  从普通落地页跳到 3D 空间化体验。  
  结论：多轮评估可触发非线性质量跃迁，而非线性微调。

- V1 全栈架构的关键是先把需求讲清。  
  Planner 把 1-4 句需求扩成完整规格，并强调产品语境，避免早期技术误判级联。  
  结论：前置语义澄清能减少后期返工。

- Generator 与 Evaluator 通过 Sprint 合约协作。  
  Generator 以 React/Vite/FastAPI/SQLite 实现；Evaluator 先对齐合同再验收。  
  结论：验收标准前置，能减少“各说各话”。

- 对比实验给出成本与质量的硬指标。  
  单 Agent：20 分钟/$9，但核心功能损坏；完整架构：6 小时/$200，功能完整可玩。  
  结论：架构化方法当前以“更高成本换可靠性”。

- V2 演进体现“随模型升级而删流程”。  
  移除 Sprint、评估器后置，DAW 案例降至约 4 小时/$124。  
  结论：Harness 不是越复杂越好，而是持续压力测试后保留有效组件。

## 三、关键概念白话版
- 上下文重置（Context Reset）  
  是什么：中途清空对话上下文，让新 Agent 接手。  
  为什么重要：避免长会话后模型疲劳和早收尾。  
  常见误区：以为“多给 token”就能替代重置。

- 结构化工件交接  
  是什么：用规格、任务、验收条目等标准化对象传状态。  
  为什么重要：让接力有统一语义，不依赖聊天细节。  
  常见误区：把“摘要几句”当作完整交接。

- 生成器-评估器分离  
  是什么：一个负责产出，一个负责独立打分与找错。  
  为什么重要：降低自评偏差，提升发现缺陷能力。  
  常见误区：让同一模型“自己夸自己再改自己”。

- 评分化主观质量  
  是什么：把审美/体验拆成可打分维度。  
  为什么重要：主观任务才能进入可优化闭环。  
  常见误区：只盯视觉风格，不看功能可用性。

- Sprint 合约  
  是什么：每轮实现前先对齐目标和验收边界。  
  为什么重要：减少实现与验收标准错位。  
  常见误区：把 Sprint 当纯项目管理术语，而非模型协作协议。

- 架构压力测试  
  是什么：模型升级后重测流程，删掉失效环节。  
  为什么重要：避免把旧时代 workaround 永久化。  
  常见误区：流程一旦有效就长期不动。

## 四、我们怎么看
- 我们认为这期最有价值的不是“多 Agent”本身，而是**把失败模式显式建模**。依据是其先定义两类根因（上下文与自评），再针对性设计角色与工件。受影响最大的是做 Agent 产品化落地的团队，而非只做 demo 的个人开发者。短期看，成本仍偏高；中期随模型能力上升，流程会继续简化。

- 我们判断“质量优先场景”会优先采用这套思路。依据是对比实验里，完整架构显著提升可用性与可修复性。  
  如果你的任务容错低（可玩性、可部署性、验收合格率优先），那么多角色+结构化交接值得投入；  
  如果你只追求低成本快速草稿，那么单 Agent 仍有性价比。

- 我们也认为评估器能力会成为下一阶段分水岭。依据是 Evaluator 已能定位具体可修缺陷（如路由顺序与状态条件问题），这直接决定闭环效率。  
  如果评估器继续从“能发现问题”进化到“能稳定给修复建议并验证回归”，那么整体成本曲线会进一步下降；  
  否则系统会卡在“发现很多问题但修不动”的中间态。

## 五、一句可直接借鉴
先不要急着堆提示词，把你的 Agent 流程改成“生成-独立评估-结构化交接-再生成”的闭环，再谈长任务稳定性与质量提升。
